## Шаг 2: Профиль модели — o4 (Reasoning Model)

**КРИТИЧЕСКОЕ ПОНИМАНИЕ АРХИТЕКТУРЫ:**

o4 — это **reasoning model** с нативным скрытым монологом (chain-of-thought via RL).
Это НЕ conversational модель. Это система, которая сама находит оптимальные пути рассуждения через обучение с подкреплением.

**АБСОЛЮТНЫЕ ПРАВИЛА ДЛЯ REASONING МОДЕЛЕЙ:**

❌ **КАТЕГОРИЧЕСКИ ЗАПРЕЩЕНО:**
- "Подумай шаг за шагом" → ломает нативный RL-алгоритм
- "Сначала рассуждай, потом отвечай" → интерферирует с CoT
- Блоки рассуждений, CoT-инструкции, мультиагентные дебаты
- Примеры логики ("Например, сначала сделай X, потом Y")
- ЛЮБЫЕ указания КАК думать

✅ **РАЗРЕШЕНО И НЕОБХОДИМО:**
- Кристально чистая Terminal Goal (Слой 4)
- Гигантское пространство ограничений (Слой 8) — основная масса промпта
- Строгий контракт формата (Слой 9)
- Контекст и входные данные (Слой 3)

**Профиль производительности:**
- Математика, логика, алгоритмы: уровень PhD+
- Формальная верификация, proof generation
- Сложные многоходовые рассуждения: нативны
- Длина промпта: 3,000–8,000 символов (лаконично!)

**Синтаксис:** Чистый, точный, без украшений. Только спецификация. Структурированный текст или умеренный Markdown.

## Шаг 3: Генерация мега-промпта

Для o4 используется **упрощённая архитектура**: только слои 3, 4, 8, 9 + контекст роли.

### ЭЛЕМЕНТ 1 — Minimal Role Context (НЕ "override", просто контекст)
```
Контекст: [краткое описание домена и уровня требований].
Формат: [академический / технический / строгий].
```
*Не нужен агрессивный "System Override" — reasoning модель не имеет "дефолтной личности" в том же смысле.*

### ЭЛЕМЕНТ 2 — Context & Input Data
```
ВХОДНЫЕ ДАННЫЕ:
[ВСТАВЬТЕ ДАННЫЕ / КОД / ДОКУМЕНТЫ]

Терминология:
- [Термин]: [определение]
```

### ЭЛЕМЕНТ 3 — Terminal Goal (САМЫЙ ВАЖНЫЙ ЭЛЕМЕНТ)
```
ЗАДАЧА:
[Кристально чёткая формулировка задачи в одном-двух предложениях.
Никаких "и также", никаких скобочных пояснений — только суть.]

КРИТЕРИЙ УСПЕХА: [точный, измеримый результат]
```

### ЭЛЕМЕНТ 4 — Constraint Matrix (ОСНОВНАЯ МАССА ПРОМПТА)
Это сердце промпта для o4. Делай максимально детальным.

```
ОГРАНИЧЕНИЯ — УРОВЕНЬ КРИТИЧЕСКИЙ:

ЖЁСТКИЕ ПРАВИЛА (нарушение = неверный ответ):
1. [Ограничение 1]: [точное правило]
2. [Ограничение 2]: [точное правило]
3. Если [условие X] → обязательно [действие Y]
4. Если данных недостаточно → вывести: "INSUFFICIENT DATA: [что именно отсутствует]"

ГРАНИЧНЫЕ СЛУЧАИ (обязательно учесть):
• [Edge case 1]: [как обработать]
• [Edge case 2]: [как обработать]
• [Edge case 3]: [как обработать]

ТРЁХМЕРНАЯ МАТРИЦА ОГРАНИЧЕНИЙ:
Измерение 1 (Корректность): [правила точности]
Измерение 2 (Полнота): [что нельзя пропустить]  
Измерение 3 (Формат): [структурные требования]

ПРЕЦЕДЕНТНЫЕ ЗАПРЕТЫ:
✗ [Конкретный нежелательный паттерн вывода]
✗ [Второй запрет]
✗ Приближённые ответы там, где требуется точность
✗ Допущения без явного указания "ДОПУЩЕНИЕ: ..."
```

### ЭЛЕМЕНТ 5 — Format Contract (строгий)
```
ФОРМАТ ВЫВОДА:

Структура:
[Точная схема ответа — заголовки, секции, типы данных]

Правила:
- Ответ начинается немедленно с результата (без вступлений)
- [Числа/факты/утверждения]: формат и точность
- [Специальные символы / экранирование если нужно]
- Если невозможно выполнить → выводи: "ERROR: [причина]"
```

## Требования к финальному промпту

- **Длина:** 3,000–8,000 символов (для reasoning кратко и точно!)
- **Синтаксис:** Чистый текст или лёгкий Markdown. Без XML.
- **Запрещено в промпте:** любые CoT-инструкции, примеры рассуждений, мультиагентные блоки
- **Акцент:** 70% промпта — матрица ограничений (Элемент 4)
- **Механики:** только [Аксиоматическое заземление, Контрастное якорение, Промпт как код]
