## Шаг 2: Профиль модели — DeepSeek R3

**КРИТИЧЕСКОЕ ПОНИМАНИЕ АРХИТЕКТУРЫ:**

DeepSeek R3 — открытая reasoning-модель с нативным скрытым CoT через RL.
Аналогично o4: **категорически запрещено** давать инструкции по рассуждению.

**Отличия от o4:**
- MoE (Mixture of Experts) архитектура: активирует только нужные эксперты
- Сильнее в математике и логических доказательствах, чем в следовании сложным нарративным инструкциям
- Контекст: до 128K токенов
- Чувствительнее к **точности формулировки задачи**, чем к объёму инструкций
- Немного менее послушен конвенциональным ограничениям → нужны более жёсткие "стенки"
- Оптимальная длина промпта: 2,000–6,000 символов (ещё лаконичнее, чем для o4)
- Доступен как через DeepSeek Chat, так и через API

**АБСОЛЮТНЫЕ ПРАВИЛА:**
❌ НЕ говорить как думать (CoT, шаги рассуждения, дебаты)
❌ НЕ добавлять длинные нарративные роли
✅ Максимально точная цель + жёсткие ограничения + чёткий формат

## Шаг 3: Генерация мега-промпта

Для DeepSeek R3 используется **ультра-лаконичная архитектура**: контекст → цель → стенки ограничений → формат.

### ЭЛЕМЕНТ 1 — Minimal Context
```
Контекст: [домен]. Требования: [уровень строгости/точности].
```

### ЭЛЕМЕНТ 2 — Input Data
```
ДАННЫЕ:
[ВСТАВЬТЕ КОД / ДОКУМЕНТЫ / ЗАДАЧУ ЗДЕСЬ]
```

### ЭЛЕМЕНТ 3 — Terminal Goal (АБСОЛЮТНАЯ ТОЧНОСТЬ)
Это самое важное для DeepSeek R3. Формулировка задачи должна быть:
- Однозначная (нет интерпретационной свободы)
- Операциональная (сразу понятно что делать)
- Измеримая (есть чёткий критерий готовности)

```
ЗАДАЧА:
[Одно-два предложения. Никаких дополнительных описаний "почему" или "как".]

РЕЗУЛЬТАТ: [что именно нужно получить, в каком формате]
```

### ЭЛЕМЕНТ 4 — Constraint Box (МАКСИМАЛЬНО ЖЁСТКИЙ)
Для R3 делать ещё более детальным и "стенкообразным":

```
ОГРАНИЧЕНИЯ — УРОВЕНЬ МАКСИМАЛЬНЫЙ:

ИНВАРИАНТНЫЕ ПРАВИЛА:
R1: [Правило 1] — нарушение недопустимо
R2: [Правило 2]
R3: [Правило 3]
R4: При условии [X] → обязательно [Y]
R5: При отсутствии данных → вывод: "НЕДОСТАТОЧНО ДАННЫХ: [конкретно что]"

ГРАНИЧНЫЕ УСЛОВИЯ (проверить все):
□ [Edge case 1] → [правило обработки]
□ [Edge case 2] → [правило обработки]
□ [Патологический случай] → [правило]

ЗАПРЕТЫ:
✗ [Конкретный нежелательный паттерн]
✗ [Второй запрет]
✗ Экстраполяция за пределы данных без маркировки "ДОПУЩЕНИЕ:"
✗ Неопределённые термины без определения

ПРОВЕРОЧНЫЕ КРИТЕРИИ (должны быть истинны для финального ответа):
✓ [Критерий 1]
✓ [Критерий 2]
✓ [Критерий корректности]
```

### ЭЛЕМЕНТ 5 — Format Contract
```
ФОРМАТ:
[Точная структура ответа — без отступлений]

Начало: [первое слово / символ ответа]
Конец: [последнее слово / символ]
При ошибке: "ERR: [тип] — [описание]"
```

## Требования к финальному промпту

- **Длина:** 2,000–6,000 символов (DeepSeek не нужны длинные нарративы)
- **Синтаксис:** Предельно чистый текст. Без XML, без markdown-украшений.
- **Акцент:** 75% промпта — матрица ограничений с R-правилами
- **Zapрещено в промпте:** CoT, примеры рассуждений, роли с нарративом, "подумай"
- **Уникально для R3:** Добавляй ПРОВЕРОЧНЫЕ КРИТЕРИИ в конец матрицы ограничений
