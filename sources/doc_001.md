Making Claude Show Its Work

Extended thinking is not a new feature. But how you use it has changed significantly with Claude 4.6.
In short
The model is moving up the stack …from stateless text generator to reasoning engine with visible internals.
What was once a black-box API call is morphing into something closer to an application primitive, one that plans, explains, and audits itself.
Extended thinking is part of a broader shift…
Models are no longer just answering questions, they are showing how they arrive at answers.
For builders, you can now inspect the reasoning chain, not just the output.
Extended thinking gives Claude a scratchpad.
Before answering, the model reasons through the problem step by step in a thinking block, then delivers the final answer in a text block.
You see both.
The model's reasoning is no longer a black box.
What changed with Claude Opus 4.6 and Sonnet 4.6 is that thinking is now summarised by default.
You get the key reasoning steps, not the raw token stream.
And there's a new adaptive thinking mode that replaces the fixed budget_tokens approach.
You tell the model how hard to think, not how many tokens to burn.
The practical impact?
Better answers on hard problems, transparent reasoning for audit and debugging. With a cleaner API for production use.
Subscribe
The importance of this
Most LLM API calls are fire-and-forget.
Prompt in, text out, hope for the best.
Extended thinking changes the contract.
The model shows its work.
For code generation, mathematical reasoning, multi-step analysis and complex decision-making … this is the difference between “the model said X” and “the model reasoned through A, B, C and concluded X.”
That matters for debugging.
It matters for trust.
And it matters for the growing number of use cases where you need to explain why an AI system made a decision, not just what it decided.

How it works
Enable thinking by adding a thinking parameter to your API call:
The response comes back as a list of content blocks.
First a thinking block with the model's reasoning.
Then a text block with the final answer.
Two parameters control thinking:
budget_tokens — Maximum tokens the model can use for reasoning. Must be less than max_tokens .
max_tokens — Total output cap including both thinking and text.
On Opus 4.6 and Sonnet 4.6, budget_tokens is deprecated in favour of adaptive thinking with an effort parameter. More on that below.
Subscribe
Summarised thinking
Here is the important change with Claude 4 models.
The API returns a summary of the full thinking process, not the raw thinking tokens.
The model still does the full reasoning internally…you're billed for the full thinking tokens…but what you see in the response is a condensed version that preserves the key reasoning steps.
Why?
Raw thinking on complex problems can run to tens of thousands of tokens.
Most of that is noise … restating the problem, considering dead ends, circling back. The summary gives you the signal.
You can control this with the display field:
"summarized" (default) — Returns condensed thinking.
"omitted" — Returns empty thinking blocks with an encrypted signature .
Useful when you only care about the final answer but need to maintain multi-turn thinking continuity.
With display: "omitted" , text streaming begins immediately after the signature...no thinking tokens transmitted. Same cost, faster time-to-first-text.

Streaming
For real-time applications, stream the thinking as it happens:
The event sequence: message_start > content_block_start (thinking) > thinking_delta events > signature_delta > content_block_stop > content_block_start (text) > text_delta events > done.

Extended thinking with tool use
This is where it gets interesting for agentic applications.
Extended thinking works alongside tool use.
The model reasons about which tool to call, processes the result, reasons again, and produces a final answer. The thinking is visible at every step.
Critical detail: when you pass tool results back to the model in a multi-turn conversation, you must include the thinking blocks from the previous assistant turn unchanged.
The model needs its reasoning history to maintain continuity.
Subscribe
Interleaved thinking
On Claude 4 models, the model can think between tool calls.
Not just once at the start.
Without interleaved thinking, the model thinks, calls a tool, calls another tool (no thinking), then answers.
With interleaved thinking: the model thinks, calls a tool, thinks again about the result, calls another tool, thinks again, then answers.
On Opus 4.6 and Sonnet 4.6 with adaptive thinking, interleaved thinking is enabled automatically. For other Claude 4 models, add the beta header interleaved-thinking-2025-05-14 .
This is significant for complex agentic workflows where the model needs to reason about intermediate results before deciding the next step.
Subscribe
Billing
You are charged for the full thinking tokens, not the summarised output.
The billed output token count will not match the response token count.
The model does the full reasoning; you see the summary; you pay for the full reasoning.
The quality comes from the full reasoning.
The summary is a convenience for your application.
Subscribe
Caching considerations
Two things to know about prompt caching with extended thinking:
Thinking blocks from previous turns are removed from context. This affects cache breakpoints.
Changing thinking parameters invalidates message cache breakpoints.
If you switch budget_tokens from 4000 to 8000 between requests, cached messages get invalidated. System prompts and tools stay cached.
For long sessions, consider the 1-hour cache duration to avoid repeated cache misses.

Practical guidance
When to use extended thinking:
Mathematical reasoning and proofs
Complex code generation and debugging
Multi-step analysis where you need to audit the reasoning
Agentic workflows with tool use where intermediate reasoning matters
Any task where “show your work” has value
When to skip it:
Simple Q&A or factual lookups
High-throughput, low-latency applications where thinking overhead isn't worth it
Tasks where you only need the answer, never the reasoning
Budget sizing:
Start with budget_tokens: 10000 for most tasks
Increase to 20000–50000 for complex mathematical or coding problems
The model uses what it needs — setting a higher budget doesn't force more thinking
